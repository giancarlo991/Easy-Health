⚠️ Instruções para a sabatina: Use este guia durante as perguntas do professor. Ele contém as explicações exatas e trechos das configurações de pipeline que nós implementamos nos repositórios Easy-Health e EasyHealthApiV2.
1. Como está estruturado o workflow do seu Backend? Estrutura

O workflow está estruturado em dois Jobs principais interdependentes rodando no runner ubuntu-latest:

  • Job 1 (unit-tests): Responsável por rodar os testes unitários (Jest) com isolamento total.
  • Job 2 (integration-tests): Responsável por rodar os testes de integração HTTP (Newman). Ele possui uma dependência declarada e só executa se o primeiro passar.

Isso segue o princípio de Fail-Fast em DevOps: não gastamos minutos de runner executando testes de rede complexos se a integridade da lógica básica estiver quebrada.

integration-tests: name: 🔗 Testes de Integração – Endpoints (Newman) runs-on: ubuntu-latest needs: unit-tests # Garante a dependência entre os jobs
2. Como vocês resolvem a dependência do banco MongoDB no pipeline? Docker Services

Não conectamos o pipeline a um banco de dados externo ou de staging. Em vez disso, utilizamos o recurso de Services do GitHub Actions.

Declaramos um serviço chamado mongodb no Job de integração. O runner cria automaticamente um container Docker a partir da imagem mongo:latest e expõe a porta padrão do MongoDB (27017:27017) para a máquina virtual de teste.

services: mongodb: image: mongo:latest ports: - 27017:27017
3. Como vocês iniciam a API e evitam concorrência com o Newman? wait-on

Esta é uma questão clássica de Race Condition (condição de corrida) em pipelines. Se disparássemos o Newman logo após o start do servidor, as requisições falhariam porque o Node ainda estaria conectando ao banco.

Nós resolvemos isso com duas etapas:

  1. Iniciamos o servidor em segundo plano (background) no terminal Linux usando o operador &.
  2. Utilizamos a biblioteca wait-on para bloquear o pipeline e realizar requisições de checagem contra a porta 3000. O pipeline só avança quando a API responder com status HTTP de sucesso (com um tempo limite de 60 segundos).
- name: 🚀 Iniciar servidor Express run: | npm start & npx wait-on http://localhost:3000/ -t 60000
4. Como funcionam os testes unitários sem banco de dados? Mocking

Nos testes unitários (feitos com Jest), mockamos (simulamos) completamente todas as dependências de rede e criptografia usando o método jest.mock().

Mockamos a biblioteca bcrypt, jsonwebtoken e o próprio modelo do Mongoose User. Nos testes do AuthService, nós definimos previamente o que as funções de banco devem retornar usando mockResolvedValue() e verificamos se elas foram chamadas com os parâmetros corretos.

// Mocks antes de instanciar a classe jest.mock('../src/models/User'); jest.mock('bcrypt'); jest.mock('jsonwebtoken'); // Simulando comportamento para login correto User.findOne.mockResolvedValue(fakeUser); bcrypt.compare.mockResolvedValue(true); jwt.sign.mockReturnValue('mocked.jwt.token');
5. Como vocês orquestram os testes do Postman no pipeline? Newman CLI

Os testes de endpoints foram escritos no Postman e exportados como um arquivo JSON de Collection. No pipeline, executamos essa coleção usando a ferramenta Newman CLI (executada via npx para não precisar instalar globalmente).

Injetamos a variável de ambiente dinamicamente na CLI usando --env-var baseUrl=http://localhost:3000. Dividimos a execução em duas etapas: primeiro validamos a pasta de Autenticação (Cadastro/Login) e depois rodamos a suíte de testes de todos os outros módulos.

run: | npx newman run tests/newman/easyhealth.postman_collection.json \ --folder "Auth – Cadastro e Login" \ --env-var baseUrl=http://localhost:3000 \ --reporters cli
6. Onde ficam guardadas as chaves e segredos do banco? Secrets & Fallbacks

Utilizamos os Secrets do GitHub para criptografar chaves de produção (como JWT_SECRET e MONGODB_URI). No pipeline, acessamos esses valores usando a sintaxe ${{ secrets.CHAVE }}.

Para garantir que o pipeline de testes local/CI funcione mesmo se as secrets não estiverem preenchidas, implementamos operadores de fallback (||) no yml apontando para instâncias locais seguras de desenvolvimento.

env: MONGODB_URI: ${{ secrets.MONGODB_URI || 'mongodb://localhost:27017/easyhealth' }}
